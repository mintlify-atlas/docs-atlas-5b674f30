---
title: "Distroless quickstart: build your first container"
description: "Build and run your first distroless container using Docker multi-stage builds, verify its cosign signature, and choose the right base image for your language."
---

This guide walks you through building a production-ready distroless container image. You will choose the right base image for your application, write a multi-stage Dockerfile, verify the image signature with cosign, and run the result locally. The entire process takes only a few minutes.

<Steps>
  <Step title="Choose your base image">
    Pick the distroless base image that matches your application's runtime requirements. Use the most minimal image that satisfies your dependencies.

    | Image | When to use |
    |---|---|
    | `gcr.io/distroless/static-debian12` | Fully static binaries with no libc dependency (Go with `CGO_ENABLED=0`) |
    | `gcr.io/distroless/base-nossl-debian12` | Applications that need glibc but not libssl |
    | `gcr.io/distroless/base-debian12` | Applications that need glibc and libssl (most Go apps with cgo, most C apps) |
    | `gcr.io/distroless/cc-debian12` | C/C++ or Rust applications that link against libstdc++ |
    | `gcr.io/distroless/python3-debian13` | Python 3 applications |
    | `gcr.io/distroless/nodejs22-debian13` | Node.js 22 applications |
    | `gcr.io/distroless/java17-debian13` | Java 17 applications |
    | `gcr.io/distroless/java21-debian13` | Java 21 applications |

    Images are published as multi-arch image indexes. To target a specific architecture, append a suffix such as `:latest-amd64` or `:latest-arm64`.
  </Step>

  <Step title="Write a multi-stage Dockerfile">
    Docker multi-stage builds are the recommended way to use distroless images. Use a full build image in the first stage to compile your application, then copy only the output artifact into a distroless image for the final stage. This keeps build tools out of your runtime image entirely.

    <Tabs>
      <Tab title="Go">
        ```dockerfile
        FROM golang:1.22 as build

        WORKDIR /go/src/app
        COPY . .

        RUN go mod download
        RUN go vet -v
        RUN go test -v

        RUN CGO_ENABLED=0 go build -o /go/bin/app

        FROM gcr.io/distroless/static-debian12

        COPY --from=build /go/bin/app /
        CMD ["/app"]
        ```
      </Tab>
      <Tab title="Python">
        ```dockerfile
        FROM python:3-slim AS build-env
        COPY . /app
        WORKDIR /app

        FROM gcr.io/distroless/python3
        COPY --from=build-env /app /app
        WORKDIR /app
        CMD ["hello.py", "/etc"]
        ```
      </Tab>
      <Tab title="Node.js">
        ```dockerfile
        FROM node:22 AS build-env
        COPY . /app
        WORKDIR /app

        RUN npm ci --omit=dev

        FROM gcr.io/distroless/nodejs22-debian13
        COPY --from=build-env /app /app
        WORKDIR /app
        CMD ["hello.js"]
        ```
      </Tab>
      <Tab title="Java">
        ```dockerfile
        FROM eclipse-temurin:17-jdk AS build-env
        COPY . /app/examples
        WORKDIR /app
        RUN javac examples/*.java
        RUN jar cfe main.jar examples.HelloJava examples/*.class 

        FROM gcr.io/distroless/java17-debian13
        COPY --from=build-env /app /app
        WORKDIR /app
        CMD ["main.jar"]
        ```
      </Tab>
      <Tab title="Rust">
        ```dockerfile
        FROM rust:1 as build-env
        WORKDIR /app
        COPY . /app
        RUN cargo build --release

        FROM gcr.io/distroless/cc-debian12
        COPY --from=build-env /app/target/release/hello-world-distroless /
        CMD ["./hello-world-distroless"]
        ```
      </Tab>
    </Tabs>

    <Note>
      Distroless images do not contain a shell. Always use the **vector form** for `CMD` and `ENTRYPOINT` — for example, `CMD ["/app"]` not `CMD /app`. The string form causes the container runtime to prefix your command with a shell (`/bin/sh -c`), which does not exist in distroless images.
    </Note>
  </Step>

  <Step title="Verify the image signature">
    All distroless images are signed by [cosign](https://github.com/sigstore/cosign) with ephemeral keys (keyless signing). Verify the signature of whichever image you are using before building your final image.

    ```bash
    cosign verify $IMAGE_NAME \
      --certificate-oidc-issuer https://accounts.google.com \
      --certificate-identity keyless@distroless.iam.gserviceaccount.com
    ```

    Replace `$IMAGE_NAME` with the fully qualified image reference you chose in step 1, for example `gcr.io/distroless/static-debian12`. See [image verification](/security/verification) for more detail.
  </Step>

  <Step title="Build and run">
    Build and run your image using the standard Docker commands:

    ```bash
    docker build -t myapp .
    docker run -t myapp
    ```

    <Tip>
      If your container fails to start and you need to inspect the filesystem or test commands interactively, switch your final stage to the `:debug` tag temporarily. The debug image includes a busybox shell you can invoke with `docker run --entrypoint=sh -ti myapp`. See [debug images](/usage/debug-images) for details. Never ship the `:debug` tag to production.
    </Tip>
  </Step>
</Steps>
