---
title: "Build containers with Docker multi-stage builds"
description: "Use Docker multi-stage builds to compile your application in a full SDK image and copy only the runtime artifacts into a minimal distroless image."
---

Docker multi-stage builds are the standard way to use distroless images. The idea is straightforward: one stage uses a full SDK image to compile and test your application, and a second stage copies only the finished binary or runtime artifacts into a minimal distroless image. The final container ships none of the build tools, compilers, or package managers — only what your application needs to run. This requires Docker 17.05 or higher.

## Language examples

Each example below follows the same two-stage pattern. The first stage (`build-env` or `build`) installs dependencies and compiles the application. The second stage starts `FROM` a distroless base image and copies in only the compiled output.

<Tabs>
  <Tab title="Go">
    Go programs compiled with `CGO_ENABLED=0` produce a fully static binary with no external library dependencies, making `gcr.io/distroless/static-debian12` the right base image.

    ```dockerfile
    FROM golang:1.22 as build

    WORKDIR /go/src/app
    COPY . .

    RUN go mod download
    RUN go vet -v
    RUN go test -v

    RUN CGO_ENABLED=0 go build -o /go/bin/app

    FROM gcr.io/distroless/static-debian12

    COPY --from=build /go/bin/app /
    CMD ["/app"]
    ```
  </Tab>
  <Tab title="Java">
    Java applications need a JRE at runtime. The distroless Java images bundle the JDK's runtime components so you only need to copy your compiled jar.

    ```dockerfile
    FROM eclipse-temurin:17-jdk AS build-env
    COPY . /app/examples
    WORKDIR /app
    RUN javac examples/*.java
    RUN jar cfe main.jar examples.HelloJava examples/*.class

    FROM gcr.io/distroless/java17-debian13
    COPY --from=build-env /app /app
    WORKDIR /app
    CMD ["main.jar"]
    ```
  </Tab>
  <Tab title="Python">
    Python applications copy the source code into the image. The distroless Python image provides the interpreter, so no installation step is needed in the final stage.

    ```dockerfile
    FROM python:3-slim AS build-env
    COPY . /app
    WORKDIR /app

    FROM gcr.io/distroless/python3
    COPY --from=build-env /app /app
    WORKDIR /app
    CMD ["hello.py", "/etc"]
    ```
  </Tab>
  <Tab title="Node.js">
    Node.js applications install only production dependencies in the build stage, then copy the entire app directory — including `node_modules` — into the distroless image.

    ```dockerfile
    FROM node:22 AS build-env
    COPY . /app
    WORKDIR /app

    RUN npm ci --omit=dev

    FROM gcr.io/distroless/nodejs22-debian13
    COPY --from=build-env /app /app
    WORKDIR /app
    CMD ["hello.js"]
    ```
  </Tab>
  <Tab title="Rust">
    Rust release builds are statically linked against musl or dynamically against glibc. This example uses `gcr.io/distroless/cc-debian12`, which includes glibc and libssl for dynamically linked binaries.

    ```dockerfile
    FROM rust:1 as build-env
    WORKDIR /app
    COPY . /app
    RUN cargo build --release

    FROM gcr.io/distroless/cc-debian12
    COPY --from=build-env /app/target/release/hello-world-distroless /
    CMD ["./hello-world-distroless"]
    ```
  </Tab>
</Tabs>

## Running the examples

Navigate to the directory for your language and run the standard build and run commands.

```bash
docker build -t myapp .
docker run -t myapp
```

<Note>
  Multi-stage builds require **Docker 17.05 or higher**. Run `docker version` to confirm your installed version before building.
</Note>

<Tip>
  If your container exits unexpectedly or you need to inspect the filesystem, switch the final `FROM` line to the `:debug` tag to get a busybox shell. See [debug images](/usage/debug-images) for details.
</Tip>
