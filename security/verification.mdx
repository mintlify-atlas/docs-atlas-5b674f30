---
title: "Verify distroless image signatures with cosign"
description: "All distroless images are signed with cosign using keyless signing. Learn how to verify image signatures to protect your software supply chain."
---

All distroless images are signed by [cosign](https://github.com/sigstore/cosign) using ephemeral keys — a technique called keyless signing. This means there is no long-lived private key stored anywhere that could be compromised. Instead, each image is signed at build time using a short-lived credential bound to a Google OIDC identity. Verifying the signature before you use an image confirms it was produced by the official distroless build pipeline and has not been tampered with.

## Verifying an image

Install [cosign](https://github.com/sigstore/cosign#installation) and run the following command, replacing the image reference with the image you want to verify:

```bash
cosign verify gcr.io/distroless/static-debian12 \
  --certificate-oidc-issuer https://accounts.google.com \
  --certificate-identity keyless@distroless.iam.gserviceaccount.com
```

The `--certificate-oidc-issuer` and `--certificate-identity` flags tell cosign exactly which identity to expect in the signing certificate. If either value does not match, verification fails and you should not use the image.

<Note>
  Verify any distroless image before use. Run this command for every image reference you pull into a build pipeline, including intermediate base images.
</Note>

## What keyless signing means

Keyless signing removes the need to manage and protect a long-lived signing key. Instead:

- **No long-lived key to protect.** Each signature uses an ephemeral key that is discarded after signing. There is no private key file to rotate, leak, or revoke.
- **Signed using Google's OIDC identity.** The distroless build pipeline authenticates with Google's OIDC provider (`https://accounts.google.com`) and signs using the service account identity `keyless@distroless.iam.gserviceaccount.com`.
- **Verifiable by anyone.** Because the OIDC issuer and identity are public, you can independently verify every signature without contacting Google or trusting a third-party key distribution system.

When you run `cosign verify`, cosign checks the Rekor transparency log to confirm the signature was recorded at the time the image was published, and validates the signing certificate against the expected OIDC issuer and identity you provide.

## Integrating verification into CI/CD

Verify images before any build step that depends on them. This prevents a tampered or unexpected image from entering your pipeline undetected.

Here is an example CI step that verifies an image before building a dependent container:

```bash
# Verify the base image signature before using it in a build
cosign verify gcr.io/distroless/base-debian12 \
  --certificate-oidc-issuer https://accounts.google.com \
  --certificate-identity keyless@distroless.iam.gserviceaccount.com

# Proceed with the build only after successful verification
docker build -t myapp:latest .
```

If `cosign verify` exits with a non-zero status, stop the pipeline. Do not proceed with the build.

<Tip>
  After verifying an image, pin to its specific digest (e.g., `gcr.io/distroless/static-debian12@sha256:...`) for reproducible builds. This ensures subsequent pipeline runs use the exact same image bytes, even if the tag is updated.
</Tip>

## The cosign.pub file

The distroless repository includes a `cosign.pub` file at the repository root. This file contains the public key that was used for traditional (non-keyless) signature verification in earlier versions of the project. For current images, use the keyless verification command shown above — the OIDC issuer and identity are the authoritative verification parameters.
